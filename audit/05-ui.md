## Z-index / display

UIs crean modales con `position: fixed; z-index: ...`. Sin coordinacion. Si se cablean varias, riesgo solapamiento.

## Estado de inicializacion de UIs (cuando se monten)

- Algunas crean propio container en `initialize()` (ej. PartyCreationUI, CharacterSheetUI).
- Otras esperan container existente y leen via `getElementById` (ej. CombatAnimations:607 lee `#combat-container`).
- Mezcla **no documentada**.

## Recomendacion

### Patron unificado propuesto

1. `index.html` declara **slots vacios** para cada pantalla:
   ```html
   <div id="screen-party-creation" class="screen hidden"></div>
   <div id="screen-combat" class="screen hidden"></div>
   <div id="screen-inventory" class="screen hidden"></div>
   <div id="screen-character-sheet" class="screen hidden"></div>
   <div id="screen-equipment" class="screen hidden"></div>
   <div id="screen-shop" class="screen hidden"></div>
   <div id="screen-save-load" class="screen hidden"></div>
   <div id="screen-main-menu" class="screen hidden"></div>
   <div id="combat-container" class="screen hidden"></div>
   <div id="fx-layer"></div>
   ```

2. `UIRouter` controla exclusion (1 screen activa, o overlay).

3. Cada UI:
   - Constructor: recibe `containerId` o root element
   - `initialize()`: monta dentro de container ya existente
   - `show()/hide()`: toggle class `hidden`
   - `isVisible()`: getter

4. Logger `tag('UI:<name>')` en show/hide.

## Conclusion F5

**12 de 13 UIs estan huerfanas en flujo principal.** Existen, tienen styles, tienen listeners, pero ni se importan ni se montan. Integration = unificar contenedores en index.html + cablear todas en `main.js initializeUIs()`.
